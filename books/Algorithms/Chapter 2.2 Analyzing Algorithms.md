# Chapter 2.2: Analyzing Algorithms（分析演算法）

**本節重點**：學習如何分析演算法的資源需求，特別是計算時間。介紹 RAM 模型作為計算模型，詳細分析插入排序的運行時間，並引入增長階數（order of growth）和 Θ 表示法的概念。

---

## 什麼是分析演算法？

**分析演算法的定義**：
- 預測演算法所需的資源

**可能考慮的資源**：
- 記憶體
- 通訊頻寬
- 能源消耗
- **最常見**：計算時間

**分析的目的**：
- 如果分析一個問題的多個候選演算法
- 可以識別最有效的演算法
- 可能有多個可行的候選演算法
- 但在過程中通常可以排除幾個較差的演算法

---

## 計算模型：RAM 模型

### RAM 模型簡介

**什麼是 RAM 模型？**

**RAM**（Random-Access Machine，隨機存取機器）模型是本書大部分內容假設的計算模型：
- 通用的單處理器計算模型
- 演算法作為電腦程式實現
- 指令一個接一個地執行，沒有並行操作

### RAM 模型的假設

**1. 時間假設**：
- **每條指令花費相同的時間**
- **每次資料存取花費相同的時間**：
  - 使用變數的值
  - 儲存到變數中
- 換句話說，每條指令或資料存取花費常數時間
- **甚至包括陣列索引**

**2. 指令類型**：

RAM 模型包含真實電腦中常見的指令：
- **算術運算**：加、減、乘、除、餘數、下取整、上取整
- **資料移動**：載入、儲存、複製
- **控制**：條件和無條件分支、子程序呼叫和返回

**3. 資料類型**：
- **整數**
- **浮點數**（用於儲存實數近似值）
- **字元**

**布林值的處理**：
- 真實電腦通常沒有 TRUE 和 FALSE 的獨立資料類型
- 通常測試整數值是否為 0（FALSE）或非零（TRUE），如 C 語言

**4. 字長限制**：
- 假設每個資料字有位元數限制
- 當處理大小為 n 的輸入時，通常假設整數由 c log₂ n 位元表示，其中 c ≥ 1 是常數
- **要求 c ≥ 1**：使每個字可以保存 n 的值，從而能夠索引單個輸入元素
- **限制 c 為常數**：使字長不會任意增長
  - 如果字長可以任意增長，我們可以在一個字中儲存大量資料並在常數時間內對其進行所有操作——這是不現實的情況

### RAM 模型的灰色地帶

**指數運算範例**：
- 問題：指數運算是常數時間指令嗎？
- **一般情況：否**
  - 當 x 和 n 是一般整數時，計算 xⁿ 通常需要對數於 n 的時間
  - 必須擔心結果是否適合電腦字
- **特殊情況**：如果 n 是 2 的精確次方
  - 指數運算通常可以視為常數時間操作
  - 許多電腦有「左移」指令，在常數時間內將整數的位元向左移動 n 個位置
  - 在大多數電腦中，將整數的位元向左移動 1 個位置等於乘以 2
  - 將位元向左移動 n 個位置等於乘以 2ⁿ
  - 因此，只要 n 不超過電腦字中的位元數，這些電腦可以通過將整數 1 向左移動 n 個位置來在 1 條常數時間指令中計算 2ⁿ

**本書的處理方式**：
- 盡量避免 RAM 模型中的灰色地帶
- 當結果小到足以適合電腦字時，將計算 2ⁿ 和乘以 2ⁿ 視為常數時間操作

### RAM 模型的限制

**記憶體階層**：
- RAM 模型不考慮當代電腦中常見的記憶體階層
- 它既不模擬快取記憶體也不模擬虛擬記憶體
- 幾個其他計算模型嘗試考慮記憶體階層效應
- 在真實機器上的真實程式中，這些效應有時很重要

**本書的處理**：
- Section 11.5 和少數問題檢查記憶體階層效應
- 但本書中的大多數分析不考慮它們
- 包含記憶體階層的模型比 RAM 模型複雜得多，因此可能難以使用
- **而且，RAM 模型分析通常是實際機器性能的優秀預測器**

## 插入排序的分析

### 為什麼需要分析？

**實際測試的問題**：

假設你在電腦上運行插入排序並計時：
- 你首先需要用真實程式語言實現它
- 測試會告訴你什麼？
  - 在你的特定電腦上運行需要多長時間
  - 對那個特定輸入
  - 使用你創建的特定實現
  - 使用你運行的特定編譯器或解釋器
  - 使用你連結的特定函式庫
  - 與電腦上同時運行的特定後台任務一起（如檢查網路上的傳入資訊）
- 如果你再次運行，甚至可能得到不同的計時結果

**實際測試的侷限性**：
- 從在一台電腦上、對一個輸入運行一個插入排序的實現中
- 你能確定什麼？
  - 如果給它不同的輸入？
  - 如果在不同的電腦上運行？
  - 如果用不同的程式語言實現？
- **答案**：不多

**我們需要的**：一種預測給定新輸入時插入排序將需要多長時間的方法

### 分析方法

**取代計時測試**：
- 我們可以通過分析演算法本身來確定它需要多長時間
- 檢查它執行偽代碼的每一行多少次
- 檢查偽代碼的每一行運行需要多長時間

**分析步驟**：
1. 首先得出運行時間的精確但複雜的公式
2. 然後使用方便的表示法提煉公式的重要部分
3. 這種表示法可以幫助我們比較同一問題的不同演算法的運行時間

### 運行時間的依賴因素

**輸入大小的影響**：
- 運行時間取決於輸入
- 排序一千個數字比排序三個數字需要更長時間

**輸入特性的影響**：
- 插入排序對於相同大小的兩個輸入陣列可能需要不同的時間
- 取決於它們已經排序的程度
- 即使運行時間可以取決於輸入的許多特徵
- **我們將專注於已被證明具有最大影響的特徵：輸入的大小**

**運行時間的定義**：
- 將程式的運行時間描述為其輸入大小的函數
- 需要更仔細地定義「運行時間」和「輸入大小」這些術語
- 還需要明確我們是在討論：
  - 引起最壞情況行為的輸入的運行時間
  - 最佳情況行為
  - 還是其他情況

### 輸入大小的定義

**取決於問題**：

1. **許多問題**（如排序、離散傅立葉轉換）：
   - 最自然的度量是輸入中的項目數
   - 例如：被排序的項目數 n

2. **其他問題**（如兩個整數相乘）：
   - 最佳度量是用普通二進制表示法表示輸入所需的總位元數

3. **某些問題**：
   - 用多於一個數字描述輸入大小更合適
   - 例如：如果演算法的輸入是圖
   - 我們通常用頂點數和邊數兩者來描述輸入大小

### 運行時間的定義

**運行時間**：
- 演算法在特定輸入上的運行時間是執行的指令和資料存取的數量
- 在 RAM 模型的框架內，如何計算這些成本應該獨立於任何特定電腦

**簡化假設**：
- 執行偽代碼的每一行需要常數量的時間
- 一行可能比另一行花費更多或更少的時間
- 但我們假設第 k 行的每次執行花費 cₖ 時間，其中 cₖ 是常數
- 這種觀點符合 RAM 模型
- 也反映了偽代碼在大多數實際電腦上的實現方式

### 插入排序的詳細分析

**帶成本的偽代碼**：

```
INSERTION-SORT(A, n)                    成本    執行次數
1  for i = 2 to n                       c₁      n
2      key = A[i]                       c₂      n-1
3      // Insert A[i] into...           0       n-1
4      j = i - 1                        c₄      n-1
5      while j > 0 and A[j] > key       c₅      Σ(i=2 to n) tᵢ
6          A[j+1] = A[j]                c₆      Σ(i=2 to n) (tᵢ-1)
7          j = j - 1                    c₇      Σ(i=2 to n) (tᵢ-1)
8      A[j+1] = key                     c₈      n-1
```

**符號說明**：
- **tᵢ**：對於每個 i = 2, 3, ..., n，表示 while 迴圈測試（第 5 行）對該 i 值執行的次數
- 當 for 或 while 迴圈以通常方式退出時（迴圈標頭中的測試結果為 FALSE）
- 測試執行的次數比迴圈主體多一次

**總運行時間**：
- 演算法的運行時間是每個執行的陳述句的運行時間總和
- 花費 cₖ 步執行並執行 m 次的陳述句對總運行時間貢獻 cₖm

**T(n) 的計算**：

```
T(n) = c₁n + c₂(n-1) + c₄(n-1) + c₅Σtᵢ + c₆Σ(tᵢ-1) + c₇Σ(tᵢ-1) + c₈(n-1)
```

---

## 最佳情況分析

### 最佳情況：陣列已排序

**情況描述**：
- 陣列已經排序
- 每次執行第 5 行時，key（原本在 A[i] 中的值）已經大於或等於 A[1:i-1] 中的所有值
- while 迴圈在第 5 行的第一次測試時總是退出

**tᵢ 的值**：
- tᵢ = 1，對於 i = 2, 3, ..., n

**最佳情況運行時間**：

```
T(n) = c₁n + c₂(n-1) + c₄(n-1) + c₅(n-1) + c₈(n-1)
     = (c₁ + c₂ + c₄ + c₅ + c₈)n - (c₂ + c₄ + c₅ + c₈)
```

**表達為線性函數**：
- 可以表達為 **an + b**，其中：
  - a = c₁ + c₂ + c₄ + c₅ + c₈
  - b = -(c₂ + c₄ + c₅ + c₈)
- **運行時間是 n 的線性函數**

---

## 最壞情況分析

### 最壞情況：陣列反向排序

**情況描述**：
- 陣列按反向排序順序排列（遞減順序）
- 過程必須將每個元素 A[i] 與整個已排序子陣列 A[1:i-1] 中的每個元素比較
- tᵢ = i，對於 i = 2, 3, ..., n
- 過程在第 5 行每次都發現 A[j] > key
- while 迴圈僅在 j 達到 0 時退出

**數學公式**：

利用求和公式（Appendix A，等式 A.2）：

```
Σ(i=2 to n) i = n(n+1)/2 - 1

Σ(i=2 to n) (i-1) = n(n-1)/2
```

**最壞情況運行時間**：

```
T(n) = c₁n + c₂(n-1) + c₄(n-1) + c₅[n(n+1)/2 - 1]
     + c₆[n(n-1)/2] + c₇[n(n-1)/2] + c₈(n-1)

     = (c₅/2 + c₆/2 + c₇/2)n²
     + (c₁ + c₂ + c₄ + c₅/2 - c₆/2 - c₇/2 + c₈)n
     - (c₂ + c₄ + c₅ + c₈)
```

**表達為二次函數**：
- 可以表達為 **an² + bn + c**，其中：
  - a = c₅/2 + c₆/2 + c₇/2
  - b = c₁ + c₂ + c₄ + c₅/2 - c₆/2 - c₇/2 + c₈
  - c = -(c₂ + c₄ + c₅ + c₈)
- **運行時間是 n 的二次函數**

---

## 為什麼關注最壞情況？

本書的其餘部分通常（但不總是）只專注於尋找**最壞情況運行時間**，即任何大小為 n 的輸入的最長運行時間。

### 三個原因

**1. 上界保證**：
- 演算法的最壞情況運行時間給出了任何輸入的運行時間上界
- 如果你知道它，那麼你就有保證演算法永遠不會花費更長時間
- 你不需要對運行時間做一些有根據的猜測並希望它永遠不會變得更糟
- **這對於即時計算特別重要**，在即時計算中，操作必須在截止日期前完成

**2. 最壞情況經常發生**：
- 對於某些演算法，最壞情況相當頻繁地發生
- 範例：在資料庫中搜尋特定資訊
  - 搜尋演算法的最壞情況通常在資訊不存在於資料庫中時發生
  - 在某些應用中，搜尋不存在的資訊可能很頻繁

**3. 平均情況通常與最壞情況一樣糟**：
- 範例：在由 n 個隨機選擇的數字組成的陣列上運行插入排序
- 需要檢查多長時間才能確定在子陣列 A[1:i-1] 中插入元素 A[i] 的位置？
- **平均而言**：
  - A[1:i-1] 中有一半元素小於 A[i]
  - 一半元素大於 A[i]
  - 因此，平均而言，A[i] 僅與子陣列 A[1:i-1] 的一半比較
  - tᵢ 約為 i/2
- **結果**：平均情況運行時間是輸入大小的二次函數，就像最壞情況運行時間一樣

### 平均情況分析的限制

**應用場景**：
- 在某些特定情況下，我們會對演算法的平均情況運行時間感興趣
- 本書將看到概率分析技術應用於各種演算法

**限制**：
- 平均情況分析的範圍有限
- 因為對於特定問題，什麼構成「平均」輸入可能不明顯
- 我們通常假設給定大小的所有輸入都是等可能的
- **在實踐中**：這個假設可能被違反
- 但有時我們可以使用**隨機演算法**（進行隨機選擇），以允許概率分析並產生預期運行時間

**後續章節**：
- Chapter 5 更深入探索隨機演算法
- 以及其他幾個後續章節

## 增長階數（Order of Growth）

### 簡化抽象的必要性

**之前的分析**：
- 使用常數 cₖ 表示每個陳述句的實際成本
- 最佳情況和最壞情況運行時間公式（等式 2.1 和 2.2）相當笨拙
- 這些表達式中的常數給了我們比真正需要的更多細節

**進一步簡化**：
- 將最佳情況運行時間表達為 an + b
- 將最壞情況運行時間表達為 an² + bn + c
- 我們忽略了實際陳述句成本，也忽略了抽象成本 cₖ

### 什麼是增長階數？

**真正感興趣的**：運行時間的**增長率**或**增長階數**

**考慮主導項**：
- 我們只考慮公式的主導項（例如 an²）
- 因為對於大的 n 值，低階項相對不重要
- 我們也忽略主導項的常數係數
- 因為常數因子在決定大輸入的計算效率時不如增長率重要

### 插入排序的增長階數

**最壞情況**：
- 當我們忽略低階項和主導項的常數係數時
- 只剩下主導項中的 n² 因子
- **因子 n² 是運行時間中最重要的部分**

**實例說明**：

假設在特定機器上實現的演算法對大小為 n 的輸入花費：
```
n²/100 + 100n + 17 微秒
```

**觀察**：
- n² 項的係數（1/100）和 n 項的係數（100）相差四個數量級
- 但一旦 n 超過 10,000，n²/100 項就主導 100n 項

**現實意義**：
- 雖然 10,000 可能看起來很大
- 但它比一個普通城鎮的人口還小
- 許多現實世界的問題有更大的輸入大小

## Θ 表示法（Theta Notation）

### 引入特殊表示法

為了突出運行時間的增長階數，我們有一個使用希臘字母 Θ（theta）的特殊表示法。

**插入排序的 Θ 表示法**：
- **最壞情況運行時間**：Θ(n²)
  - 發音：「theta of n-squared」或「theta n-squared」
- **最佳情況運行時間**：Θ(n)
  - 發音：「theta of n」或「theta n」

### Θ 表示法的非正式理解

**目前的理解**：
- 將 Θ 表示法視為「當 n 很大時大致成正比」
- **Θ(n²)**：當 n 很大時大致與 n² 成正比
- **Θ(n)**：當 n 很大時大致與 n 成正比

**注意**：
- 本章中非正式使用 Θ 表示法
- **Chapter 3 將精確定義它**

### 比較演算法效率

**一般原則**：
- 如果一個演算法的最壞情況運行時間具有較低的增長階數
- 我們通常認為它比另一個演算法更有效

**常數因子和低階項的影響**：
- 對於小輸入，運行時間具有較高增長階數的演算法可能比運行時間具有較低增長階數的演算法花費更少時間
- **但在足夠大的輸入上**：
  - 最壞情況運行時間為 Θ(n²) 的演算法在最壞情況下花費的時間少於最壞情況運行時間為 Θ(n³) 的演算法
  - 無論 Θ 表示法隱藏的常數如何

**交叉點**：
- 總有某個數字，比如 n₀
- 使得對於所有輸入大小 n ≥ n₀
- Θ(n²) 演算法在最壞情況下擊敗 Θ(n³) 演算法

### 實例比較

**範例公式**：
```
演算法 A: n²/100 + 100n + 17 微秒
演算法 B: 某個 Θ(n³) 演算法
```

**分析**：
- 雖然係數差異很大（1/100 vs 某個可能較小的常數）
- 但存在一個 n₀，當 n > n₀ 時
- n² 的增長率始終戰勝 n³ 的增長率
- 即使 n³ 項有非常小的係數

### 2.2-2（選擇排序）

考慮通過以下方式排序儲存在陣列 A[1:n] 中的 n 個數字：
1. 首先找到 A[1:n] 的最小元素並將其與 A[1] 交換
2. 然後找到 A[2:n] 的最小元素並將其與 A[2] 交換
3. 然後找到 A[3:n] 的最小元素並將其與 A[3] 交換
4. 對 A 的前 n-1 個元素繼續這種方式

**要求**：

a) **編寫偽代碼**：為這個演算法（稱為選擇排序）編寫偽代碼

b) **循環不變式**：這個演算法維護什麼循環不變式？

c) **n-1 次迭代**：為什麼它只需要對前 n-1 個元素運行，而不是對所有 n 個元素？

d) **最壞情況運行時間**：用 Θ 表示法給出選擇排序的最壞情況運行時間

e) **最佳情況**：最佳情況運行時間有更好嗎？

## 總結

**本節學習內容**：

1. **RAM 模型**：
   - 理解用於分析演算法的標準計算模型
   - 每條指令和資料存取花費常數時間
   - 包含真實電腦中常見的指令類型

2. **插入排序的詳細分析**：
   - 學習如何計算精確的運行時間公式
   - 最佳情況：Θ(n) —— 線性時間
   - 最壞情況：Θ(n²) —— 二次時間

3. **最壞情況分析的重要性**：
   - 提供運行時間的上界保證
   - 最壞情況經常在實踐中發生
   - 平均情況通常與最壞情況一樣糟

4. **增長階數**：
   - 真正重要的是運行時間如何隨輸入大小增長
   - 對於大輸入，主導項決定性能
   - 常數因子和低階項相對不重要

5. **Θ 表示法**：
   - 提供簡潔的方式來描述運行時間的增長率
   - 幫助比較不同演算法的效率
   - Chapter 3 將提供精確定義
